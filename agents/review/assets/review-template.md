
# Essential reproducibility checklist for authors in the *Biometrical Journal*

Congratulations on your successful submission to the *Biometrical Journal*. At this point, we ask you to carefully read through our detailed [*Reproducible Research (RR) checklist*](https://onlinelibrary.wiley.com/pb-assets/assets/15214036/ag/Reproducible-Research-Author-Checklist-2025-1761054289830.pdf) to make sure that your submission follows best practices for reproducible scientific computing. Please check the following items to certify that you have addressed them in your code submission, or that they do not apply to your code. These are essential, and your reproducibility editor will ask you for an explanation of any missing checkmark.

- [ ] **Include a `README` file:** Either `README.txt`, `README.md` or `README.pdf`, following the specifications in the *RR guideline* linked above. Please do not forget to add version and dependencies information by, e.g., copy-pasting the output of R's `sessionInfo()` *after* loading all necessary packages.
- [ ] **Describe how to run your code:** Your `README` must describe exactly how to run your code, i.e. which files need to be run in which order. Ideally, also create *one (!) main file* (e.g., `main.R` or `main.py`) that runs your script files and recreates all results of your article, i.e., tables, figures, and numbers.
- [ ] **Include all code and data necessary to reproduce all figures, tables and results in your paper:** If at all possible, the original data should be made available to RR editors for the strict purpose of the audit. If data are not allowed to be made public, pseudo data comparable to the original data in size and structure can be used instead.
- [ ] **Ensure that the results from your code are clearly linked to the results in your paper:** Figures and tables in the same format as in the article should be exported to appropriately named files (e.g. table1.csv, figure1.pdf), or clearly labelled and displayed in a results notebook.
- [ ] **Ensure simulations are reproducible:** All code that relies on results of a random number generator must be initiated by seeds so that results are reproducible.
- [ ] **Report the runtime of your code, and what hardware you used in your `README`:** The order of magnitude is usually sufficient, i.e. make clear if the code runs for a couple minutes, a few hours, or many days, and what hardware it was executed on.
- [ ] **Explain the use of parallelisation:** If your code runs in parallel, explicitly state so in the `README` and describe how it can be adapted to the users' own set-up.
- [ ] **Avoid absolute paths:** Use paths relative to the current file, or a working directory if the latter is specified in the `README` and script files (e.g., `./script.R`, or `data/`). If this is impossible, clearly state in your `README` how to change the paths. Refrain from setting or changing the working directory programmatically in your code.
- [ ] **Organise your code files well:** Use sensible folder structures, sensible file names, sensible variable names, and avoid code duplication by the use of functions.
- [ ] **Add helpful comments:** In particular add comments that state what figure or table in the manuscript the code produces and make it as easy as possible to associate the outputs of your code with the results in the manuscript.
- [ ] **Provide intermediate results:** If the computations run for more than a couple of hours on a standard desktop PC, please send along intermediate results. When doing that, ensure that each simulation replication is independently reproducible (e.g., by using a different seed for each replication), or alternatively, provide results for a reduced number of repetitions that allow to extrapolate reproducibility for the full number of repetitions. This facilitates spot checks of reproducibility without having to re-run your entire simulation study. If the intermediate results are too large to share via e-mail, please upload them to a repository such as, e.g., [*zenodo*](https://about.zenodo.org/), [*figshare*](https://figshare.com/) or [*osf.io*](https://osf.io/).
- [ ] **All code files, data and the `README` should be contained in a single .zip container:** E.g. `Code_and_Data.zip`, with subfolders in the container to organize the codebase. Strip that `.zip` container of any unnecessary files.
